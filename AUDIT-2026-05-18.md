## 3. Architecture & Design Assessment

### Macro Overview (actual, not idealized)

- **SPA owns no business logic.** Every state-changing action goes through `apps/web/src/lib/apiClient.ts` against `/functions/v1/<bundle>/...`. SPA never directly PostgREST-writes. Only the auth/login flow touches PostgREST directly.
- **Domain logic lives in three places, in this hierarchy:**
  1. **Postgres triggers** for derived state and journal entries â€” `0060_je_auto_generation_hooks.sql` (six AFTER triggers post JE rows); rollups maintain invoice header totals + PO totals (`0058`); stock movements (`0061`/`0063`); period-close enforcement (`0074`); audit rows (`0041`/`0047`/`0058`/`0068`).
  2. **Thin edge handlers** do request/response shaping, idempotency, state-machine assertion, and a handful of SECURITY DEFINER RPCs (`convert_quote_to_invoice`, `convert_lead`, `provision_organization`, `next_doc_number`, `post_journal_entry`).
  3. **SPA** is presentational + React Query cache.
- **Wire envelope is uniform**: `{ data }` / `{ error: { code, message, request_id } }` from `_shared/responses.ts:63`, parsed back by `apiClient.ts:41`. Every non-GET is idempotency-keyed at the client (`apiClient.ts:84`) and replay-cached on the server (`handler-helpers.ts:143`).
- **Multi-tenancy is a JWT claim â†’ SQL helper.** `app_metadata.team1_org_id` flows into `public.current_org_id()` (used by RLS) and into `caller.orgId` (used by every `.eq('org_id', â€¦)` in handlers). Belt-and-suspenders by design. **However, see [Security C-1] â€” the SQL side reads the wrong claim.**

### Strengths

- **Genuinely consistent edge-function template.** Every bundle is `index.ts` â†’ `route.ts` â†’ `handlers/*.ts` â†’ `schemas.ts`, with `_shared/handler-helpers.ts` providing `parseBody/parseLimit/encodeCursor/paginate/respondWithIdempotency/admin/requireCap`. 22 bundles all look the same; new handlers cost ~80 LOC. `invoicing-api/routes.ts` and `invoicing-api/index.ts` are 31 + 100 lines respectively â€” the boilerplate is gone.
- **Capability matrix is rule-driven, not enumerated.** `_shared/capabilities.ts:191 allow()` is a single switch keyed on role with prefix tests; adding a cap is a one-line addition to `ALL_CAPABILITIES`. The SPA mirror at `apps/web/src/lib/capabilities.ts` byte-mirrors the same `allow()` logic.
- **State machines are real and parity-tested.** 13 machines defined once in `_shared/workflow.ts` (463 LOC), byte-mirrored to `apps/web/src/lib/workflow.ts`. `assertTransition('invoice', existing.status, 'pending')` is called *before* the UPDATE â€” defense against malformed transitions. A parity contract test enforces it.
- **Derived state has one home: DB triggers.** JE generation (0060), PO/VB/expense totals (0058), invoice rollups (0041/0052), stock movements (0063), audit-state-change rows (0041/0047/0058) â€” every non-trivial derivation lives in a trigger, with handlers writing only inputs. This is a rare and disciplined pattern at this scale.
- **Numbering is unified.** Migration 0034 + 0065 collapsed per-table sequence helpers into one `next_doc_number(org_id, doc_type)` RPC; the `_shared/numbering.ts:47` wrapper is the only client. PR #78 dropped legacy helpers.
- **Money as integer cents end-to-end** is faithfully implemented: bigint columns, `roundHalfEven` mirrored in `_shared/money.ts` + `lib/money.ts`, no float arithmetic in the data path.

### Concerns

- **[H] Handler files are getting long and copy-shaped.** `invoicing-api/handlers/invoices.ts` (836 LOC), `quotes-api/handlers/quotes.ts` (832), `finance-api/handlers/journal-entries.ts` (757), `invoicing-api/handlers/payments.ts` (626), `invoicing-api/handlers/credit-notes.ts` (504). The list/get/create/patch/transition pattern is repeated verbatim in each (`invoices.ts:201`, `:260`, `:275`, `:343`, `:409`). A generic `defineEntityHandlers({ table, cols, schema, capRead, capWrite, machine })` factory would compress ~3-4k LOC. Severity High because every new entity replays the same ~600 LOC.

- **[H] `requireOrgContext` decodes JWT without verifying signature.** `_shared/tenant.ts:34 decodeJwtPayload` does `atob(parts[1])` and parses. The comment says "the Supabase gateway has already verified" â€” true *when* `verify_jwt = true` is set in `supabase/config.toml`, but it depends on a config setting outside this file. If a future bundle ships with `verify_jwt = false` (already true for some webhook-style functions), tenant impersonation is one header away. No defense-in-depth verification. Severity High; cheap to fix with a per-bundle assert.

- **[M] Capability matrix duplicated, not shared.** `_shared/capabilities.ts` (Deno, 340 LOC) and `apps/web/src/lib/capabilities.ts` (Node, 122 LOC) are *manually* kept in step. The doc comment at `apps/web/src/lib/capabilities.ts:13` says "paired edit." There is no codegen, no shared module â€” the SPA file doesn't even enumerate `ALL_CAPABILITIES` (it only mirrors `allow()`). The contract test pins structure, not membership. A drift here means a missing button or a spurious one. The SPA mirror is *already* missing the `vendor_user` case enumerated in `_shared/capabilities.ts:283-293`. Severity Medium because the server is still authoritative.

- **[M] `types.ts` is a 2780-line god-file**, byte-mirrored across boundaries. `apps/web/src/lib/types.ts` and `supabase/functions/_shared/types.ts` are both 2780 LOC of Zod schemas for every entity, request, response, version, mini-projection. Single file means every contributor merges into the same file (collision risk during parallel waves â€” analogous to the migration slot collision problem). Splitting by domain (`types/invoicing.ts`, `types/crm.ts`) would be a low-risk win.

- **[M] queryKeys is a flat 36-file folder.** `apps/web/src/lib/queryKeys/{auth,chartOfAccounts,collaboration,contacts,creditNotes,crm,customers,â€¦}.ts`. Each file has the same `{ all, list, detail, â€¦ }` shape (`queryKeys/invoices.ts` is 13 LOC). 36 imports per consumer is awkward; co-locating with services or generating from a `defineEntityKeys()` factory would compress this.

- **[M] Numbering helper hard-codes the closed `DocKind` enum.** `_shared/numbering.ts:26` enumerates 12 doc kinds. `numbering_sequences` is a per-org table â€” adding a new doc kind needs both a code edit and a per-org seed row insertion. Not a bug, but tenants who want custom doc types are blocked at the type system.

- **[L] `parseLimit` and pagination contract is implicit.** `handler-helpers.ts:76` defaults to 50, clamps to 200. The keyset cursor format `(created_at, id)` is hard-coded in `encodeCursor`/`decodeCursor` (lines 98-114), and the SPA at `apiClient.ts` doesn't model pagination â€” every service handles cursors ad-hoc (`invoicesService.ts:43 cursor?: string`). Workable but not testable as a shared contract.

- **[L] `routes.tsx` is 1043 lines of repetition.** Every route wraps a lazy page in `<ProtectedRoute>` by hand (`routes.tsx:209-1028`). A `{ path, page, flag?, guard? }` array driving `<Route>` map would cut this by ~80% and make adding routes a config change.

- **[L] Audit-log writes are dual-path.** Triggers emit state-change rows (0041/0047/0058/0068) AND handlers call `writeAudit()` for create/update/soft-delete (`_shared/audit.ts:84`). The two-source nature is documented (`audit.ts:13-17`), but readers debugging "why two rows for one event" will hit this.

- **[L] No anti-corruption layer between the DB shape and the API shape.** `INVOICE_COLS` at `invoicing-api/handlers/invoices.ts:73-79` is a hand-written column list; `INVOICE_VERSION_COLS` at line 81; equivalent string literals in every handler. A schema-generated `select()` (from `database.types.ts`) would prevent the drift that already bit Wave 5 (`customer_name_snapshot` vs `customer_name`, captured at `invoices.ts:28-34`).

### Cross-cutting architectural risks

- **Forward-only migrations + hand-numbered slots** are still a parallel-dispatch tax. 74 migrations, two gaps at 0021/0022, repeated rename/cleanup migrations (0044, 0054, 0066, 0067) suggest a fair bit of in-flight schema churn that becomes irreversible once on cloud.
- **`admin()` everywhere = no RLS coverage on the write path.** Every edge handler uses service-role (`handler-helpers.ts:176`). RLS is real but only protects (a) direct PostgREST reads from the SPA (which doesn't exist â€” only the auth + login flow touches PostgREST) and (b) cross-tenant defense-in-depth if a handler forgets `.eq('org_id', â€¦)`. A single missed filter = cross-tenant leak. The nightly RLS probe is the safety net. The 280 `.eq('org_id'` occurrences across 64 files are the cost.
- **No event bus / no async work queue** outside `notifications-worker` and `pdf-worker`. Anything cross-bundle (e.g. "when invoice is sent, fire pdf-worker") happens via DB triggers + pg_cron. Workable at current scale; will need rethinking before multi-region or webhooks-to-customers.
- **SPA bundle ~50 kB gzip** is excellent given 455 .ts/.tsx files, made possible by aggressive route-level `lazy()` (every page in `routes.tsx` is lazy). The 2780-line `types.ts` is shipped to the client though â€” that's all Zod schemas at runtime. Tree-shaking should drop unused schemas; worth verifying that's actually happening.

### Top 3 Architecture Recommendations

1. **Extract a `defineCrudHandlers()` factory in `_shared/`.** Most of the ~7500 LOC across `invoicing-api/handlers/`, `quotes-api/handlers/`, `finance-api/handlers/` is list-get-create-patch-transition boilerplate around a (table, cols, schema, machine, caps) tuple. Factoring this would compress code, force a single pagination contract, and make the "DB-wins" column-list drift one place to fix. Highest leverage refactor.
2. **Verify-not-just-decode the JWT in `requireOrgContext`.** Add an HMAC/JWKS verify step (cheap with `jose`) so the trust boundary doesn't depend on `verify_jwt = true` in `supabase/config.toml`. Defense in depth against future bundle config drift.
3. **Split `types.ts` by domain and codegen the capability matrix.** A `types/{crm,invoicing,finance,inventory,â€¦}.ts` layout plus a `capabilities.gen.ts` (generated from `_shared/capabilities.ts`) would eliminate two of the largest "remember to keep these byte-aligned" rules and remove a class of subtle drift bugs (e.g. the missing `vendor_user` case noted above).

## 5. Security & Compliance Issues

### Critical

- **[C-1] `current_org_id()` reads the WRONG JWT claim â€” RLS defense-in-depth dead for multi-org users** â€” `supabase/migrations/0029_tenancy_init.sql:283`. The function reads `current_setting('request.jwt.claims', true)::jsonb ->> 'org_id'` (root-level `org_id`), but the entire app stamps `app_metadata.team1_org_id` (see `auth-api/handlers/switch-org.ts:67-69`, `_shared/tenant.ts:77`). I grepped every migration: `'team1_org_id'` appears in **zero** migrations. **Result:** the JWT-claim branch of `current_org_id()` always returns NULL, so the function falls through to `SELECT om.org_id FROM org_memberships ... LIMIT 1` â€” with no `ORDER BY`. For a user with memberships in multiple orgs, RLS will pin them to an arbitrary org chosen by Postgres, not the org they switched to.

  **Impact:**
  - (a) The RLS defense-in-depth layer is effectively dead for multi-org users â€” anything reaching Postgres via the `authenticated` role (Supabase Storage attachments policies, any direct PostgREST read from the SPA, future RPCs that don't use service_role) can leak data from the "wrong" org of the same user.
  - (b) `is_staff()` / `current_user_role()` / `current_user_customer_id()` all chain off `current_org_id()` and inherit the bug.
  - (c) Storage attachments RLS (see H-2) depends directly on this.

  Today most reads go through service_role + handler-side `.eq('org_id', caller.orgId)`, so it isn't actively exploited at scale (single user = single org for both prod tenants). But it's a **one-bug-from-breach posture**, not defense-in-depth.

  **Fix direction:** rewrite `current_org_id()` to read `(current_setting('request.jwt.claims', true)::jsonb -> 'app_metadata' ->> 'team1_org_id')::uuid` with the same exception envelope, then re-run the nightly RLS probe.

### High

- **[H-1] PostgREST `or()` string injection via search-style `q` parameter** â€” widespread, e.g. `supabase/functions/invoicing-api/handlers/invoices.ts:231-234`, `search-api/handlers/search.ts:190`, `quotes-api/handlers/quotes.ts:262`, `projects-api/handlers/projects.ts:158`, and ~25 other `.or(...)` call sites. The pattern escapes `\`/`%`/`_` for ILIKE-wildcard safety, but does NOT escape `,` `(` `)` or `.` â€” and those are the PostgREST `or` mini-grammar's separators. User-controlled `q` is interpolated into an `or()` string like:
  ```ts
  `invoice_number.ilike.${like},customer_name_snapshot.ilike.${like}`
  ```
  A `q` value containing `,column.eq.value` extends the `or` with arbitrary additional predicates of the attacker's choosing.

  **Impact:** within the org-scoped RLS surface (the outer `.eq('org_id', orgId)` survives), an authenticated user can craft `q` to add boolean filters that probe field values (e.g. extracting precise totals, statuses, IDs they would otherwise have to read in bulk). Doesn't escalate org boundary, but is a confidentiality issue against fine-grained authorization assumptions and a stepping-stone if combined with C-1.

  **Fix direction:** add a `,()` to the escape regex in every handler, OR (better) factor a `safeOrIlike(columns, q)` helper in `_shared/handler-helpers.ts` that emits a single canonical form, then lint-ban inline `.or(...)`.

- **[H-2] Storage RLS depends on broken `current_org_id()`** â€” `supabase/migrations/0069_phase16_collaboration.sql:359-389`. The `attachments` bucket's `storage.objects` policies enforce `(storage.foldername(name))[1] = public.current_org_id()::text`. Because of C-1, this collapses to "the org_id of the user's *first* membership row." A multi-org user can call Supabase Storage directly (no edge function in front) and read attachments from a sibling org.

  **Impact:** direct attachment-bucket access bypasses every handler-side `.eq('org_id', caller.orgId)`. Mitigated today only because prod users are single-org.

  **Fix direction:** auto-fixes once C-1 is fixed; no separate change needed if you also add an RLS probe asserting hostile multi-org scenarios.

### Medium

- **[M-1] Notifications-worker shared-secret comparison is timing-attack-vulnerable** â€” `supabase/functions/notifications-worker/handlers/drain.ts:51`. Uses `provided !== expected` (early-out string compare). Should use a constant-time compare. The endpoint is internet-exposed (no `verify_jwt`) and the secret is the only auth gate; an attacker who can reach the URL can theoretically time-leak the secret byte-by-byte. Realistically Deno Deploy's noisy network jitter masks single-byte timing, but the pattern is wrong. **Fix:** SHA-256 both sides and compare hashes, or use a CSPRNG-derived HMAC of a per-request nonce.

- **[M-2] No CSP, no HSTS** from Vercel â€” `vercel.json:24-31`. Sets X-Frame-Options/X-Content-Type-Options/Referrer-Policy/Permissions-Policy but no `Content-Security-Policy` and no HSTS. The SPA loads only from `ts-1-lime.vercel.app` and same-origin Supabase functions â€” a CSP would meaningfully shrink XSS blast radius. HSTS is essential pre-launch for any custom domain. **Fix:** add `Strict-Transport-Security: max-age=63072000; includeSubDomains; preload` and a CSP that pins `connect-src` to the Supabase project + `script-src 'self'` (no `unsafe-inline`/`unsafe-eval`).

- **[M-3] `pg_cron` notifications-worker uses `current_setting('app.notifications_worker_secret', true)`** â€” `supabase/migrations/0070_phase19_pdf_email.sql:192`. Per the project's own `feedback_ts1_pg_cron_shared_secret.md` lesson, service_role cannot set DB-level GUCs, so `current_setting('app.notifications_worker_secret', true)` returns the empty string in production unless something *outside* this migration injected the secret. If empty, the cron POST sends `X-Worker-Secret: ''`, which `drain.ts:51` correctly rejects (`!provided` is true â†’ 401) â€” so worst case it's a silent denial-of-feature (notifications stop), not an open auth bypass. But the memory references a Vault-based pattern (`vault.create_secret` + `vault.decrypted_secrets`) that should be wired in instead. **Fix:** switch to the Vault pattern documented in memory and add a smoke test that drains return 200.

- **[M-4] `tenants-api/resolve-host` is unauthenticated and unrate-limited** â€” `supabase/functions/tenants-api/handlers/resolve-host.ts:27`. Public endpoint enumerable for "does this hostname map to an active org" â€” an attacker can scrape org slugs cheaply. Suspended orgs return NOT_FOUND (good) but live orgs return real branding metadata. **Fix:** add Supabase Edge Function rate-limit middleware (per IP) or, if not feasible, accept and document the tradeoff.

### Low / informational

- **[L-1] Supabase URL + anon key hardcoded as fallback** in `apps/web/src/lib/supabase.ts:10-12` â€” anon keys are public by design, so this is not a leak. But it means a misconfigured Vercel deployment will still talk to *production* Supabase, which is the opposite of safe-by-default. Consider removing the fallback in prod builds or making it environment-aware.

- **[L-2] `idempotency_keys` PK is `(key, user_id)` not `(key, user_id, org_id)`** â€” already deliberately deferred per CLAUDE.md "Wire envelope + idempotency" (R-W1-04). The org-mismatch path is handled defensively in `idempotency.ts:148-151`.

- **[L-3] `'org_id'` in `current_org_id()` strongly suggests the function predates the `team1_org_id` claim convention** â€” a forward-fix should also delete the old fallback membership-lookup path entirely, because once the claim is mandatory, a NULL claim means "user genuinely has no active org" and the right response is RLS-zero-rows, not silently impersonating their first membership.

- **[L-4] `console.warn` of error envelopes** â€” `_shared/idempotency.ts:199`. Persist failures log only via `console.warn` not the structured logger. Minor; rotate to `logger.warn`.

- **[L-5] Audit-log triggers vs handler writes:** the codebase enforces audit via DB triggers for state changes and handler-side `writeAudit()` for create/edit â€” a thoughtful separation. No issues observed.

### Strengths (defensive controls observed)

- **Single service-role client factory** at `_shared/supabase-admin.ts`; lint-enforced per architecture Â§2.4. Verified by grep â€” no `createClient(...)` elsewhere except in tests.

### Performance Issues

- **[M] N+1 in `replaceQuoteLines` / `appendQuoteLine` / `patchQuoteLine`** â€” `supabase/functions/quotes-api/handlers/line-items.ts:269-292` (and mirror in `invoicing-api/handlers/line-items.ts`). `resolveTaxRate` is called **once per line** inside the `for` loop. For a 50-line quote replace, that's 50 sequential `taxes.select('rate')` round trips before the bulk INSERT. Fix: pre-fetch all distinct `tax_id`s in a single `.in('id', taxIds)` call and build a Map.

- **[M] `reorderQuoteLines` does 2Ă—N sequential UPDATE round trips** â€” `quotes-api/handlers/line-items.ts:517-543`. For a 30-line quote that's 60 sequential UPDATEs to the edge. The "negative-space shift" is defensive against a uniqueness constraint that "isn't one today" (per its own comment). Could be one `update().in('id', ids)` per phase, or a single SECURITY DEFINER RPC that does the swap server-side in one call.

- **[M] `notifications-worker` drain is serial with auth-admin lookup per row** â€” `notifications-worker/handlers/drain.ts:121-167`. For batch of 50: 50 sequential `sb.auth.admin.getUserById` + 1-2 UPDATEs per row + 1 email send. `getUserById` is a remote auth-admin call (~150-300ms each); cold + hot path can take 30+ seconds. Fix: batch resolve emails via `auth.users` JOIN in one query (the worker runs as service-role so it can read auth.users directly via SQL), then parallelize the send with `Promise.all` in chunks of N.

- **[M] `recomputeQuoteTotals` is handler-side, not trigger** â€” `quotes-api/handlers/line-items.ts:154-195`. Comment claims the DB trigger handles invoices; for quotes the handler does it after every mutation. Race: two concurrent line edits can both read the rows, both compute, and the second overwrites with stale totals (no row-version, no SELECT-FOR-UPDATE). Move to an AIUD trigger like invoices have, or guard with `pg_advisory_xact_lock(quote_id)`.

- **[L] `quote_line_items` lacks the `(quote_id, position)` index** advertised by the order-by in `listQuoteLines` (`line-items.ts:221-222`). At scale this becomes a sort. Confirm with `EXPLAIN`; if it's not there, add `CREATE INDEX ... ON quote_line_items (quote_id, position)`.

- **[L] Two `unify_numbering` migrations** (0034 then 0065) â€” both define `next_doc_number`. 0065 is the canonical version; 0034 leaves an inferior copy in pg_proc until 0065 replaces. Not a perf issue; documentation hygiene only.

- **[L] `recompute_purchase_order_totals` runs the whole SUM(po_line_items) per row mutation** (`0058:87-90`). On 100-line POs that's an aggregation per line edit. Acceptable today (POs are short); flag if line-item count grows.

- **[L] Migration 0030 (moneyâ†’cents) does an unbatched `UPDATE â€¦ SET subtotal_cents = round(...)` across every money-bearing table** â€” fine for the empty/near-empty TS1 prod at the time it ran (per its inline comment "cloud was still fresh"), but the same pattern in a future re-application against a real-data DB would do full-table rewrites + drift-check scans without a `WHERE` filter or batching. Forward-only â€” flag only for future schema work.

- **[L] `JE auto-generation triggers (0060)` fire on every invoice state change + every payment INSERT + every CN allocation INSERT.** Each trigger calls `post_journal_entry` which inserts JE header + lines and runs `check_journal_balance`. This is constitutional ("triggers, not handlers"), but at scale (thousands of payments/day) consider batching balance checks.

- **[L] `pdf-worker` cold-start risk is acknowledged in-code** (`pdf-worker/index.ts:11`: 3-5s on first request after isolate spin). Mitigation in place (30s timeout, retryable hint). Consider a warmup ping if perceptible to users.

### Reliability Issues

- **[M] `withIdempotency` upsert failures are swallowed via `console.warn`** (`_shared/idempotency.ts:196-200`). If the cache write fails, the user sees success, but a retry with the same key hits the empty cache and replays the *real* mutation. For non-idempotent business effects (sending an email, charging a payment) this is the canonical double-spend risk. Promote to `logError` at minimum so we can detect; ideally fail the request.

- **[M] No dead-letter strategy on `notifications-worker`.** Failures land in `notifications.failed_at + failure_reason` and never retry (`drain.ts:121-167` only picks up rows where both `delivered_at IS NULL AND failed_at IS NULL`). Transient SMTP failures = permanent dead notification. Add a retry counter with exponential backoff + dead-letter beyond N.

- **[M] No optimistic locking / version column on quotes, invoices, projects.** Header columns (subtotal_cents, total_cents, status) can race when two clients edit the same quote. The state-machine gates state changes (draftâ†’sent is one-shot) but field edits on a draft race freely. Consider an `updated_at`-as-version `WHERE updated_at = ?` guard on PATCH paths.

- **[L] `respondWithIdempotency` strips CORS headers from the cached replay.** Comparing with `responses.ts::ok`, the wrapper at `handler-helpers.ts:162-172` builds its own header set without `corsHeaders(req)`. Non-replay paths build through `ok()`/`err()` and DO include CORS. SPA on a different origin retrying with same Idempotency-Key gets the replay without CORS â†’ browser blocks.

- **[L] `period_close` lock is implemented via the `tg_journal_entries_reject_closed_period` trigger (Wave 11C).** Correct, but means every JE INSERT does a `period_close` lookup. Confirmed reasonable; period_close should be a small table.

- **[L] Error handling is consistent** â€” every bundle index wraps `route()` in a try/catch that maps `ApiError â†’ fromApiError` (status/code mapping) and everything else to 500. `request_id` is echoed on every response (`responses.ts:48-61`).

### Top 5 Perf/Reliability Priorities

1. **Fix `replaceQuoteLines` tax-rate N+1** (`quotes-api/handlers/line-items.ts:269-276` + mirror in `invoicing-api`). Pre-fetch distinct tax_ids in one `.in('id', ...)` call. Easy win, hot path.
2. **Move quote totals recompute into an AIUD trigger** (mirror invoices). Eliminates race between concurrent line edits and a handler-side query.
3. **Promote `idempotency` upsert-failure from `console.warn` to a real error path** â€” silent persist failures here are the kind of bug that detonates in production once and you spend a week finding it.
4. **Batch the notifications-worker drain.** Resolve all 50 recipient emails in one SQL query, then `Promise.all` the email sends. Add retry-with-backoff so transient SMTP doesn't dead-letter the row.
5. **Replace the 2Ă—N reorder loop with a single `update().in()` or a server-side reorder RPC.** Same shape as `replace_quote_line_items` already in 0008.

## 10. Cross-Cutting / Systemic Issues

1. **Doubled-source-of-truth files require manual byte-parity.**

   - `_shared/workflow.ts` â†” `apps/web/src/lib/workflow.ts`

   Contract tests catch *structural* parity (Zod `_def` equality, workflow matrix equality), not *behavioral* parity. A build-time copy step (single source, generated mirror) would close this class of bug entirely.

2. **Forward-only migrations + hand-numbered slots** are a known parallel-dispatch tax. Two gaps at 0021/0022; repeated rename/cleanup migrations (0044, 0054, 0066, 0067) suggest schema churn becoming irreversible once on cloud.

3. **`admin()` everywhere = no RLS coverage on the write path.** Every handler uses service-role + manually re-applies `.eq('org_id', caller.orgId)` (280 occurrences across 64 files). A single missed filter = cross-tenant leak. The nightly RLS probe is the only structural safety net â€” and **C-1 means the probe's defense-in-depth promise is partially false today** for multi-org users.

4. **Documentation lives in two homes.** Operational lore (closeouts, retros, feedback memos) lives in `TS1/03-workspace/` and the user's memory layer, *outside* `app/`. Anyone cloning the GitHub repo alone misses ~80% of the institutional knowledge.

5. **CI gates the cheap things, skips the expensive things.** Typecheck/lint/unit/contract/build/bundle-budget all gate PRs. E2E, RLS probe, Lighthouse, coverage all run nightly (or not at all). The most security-critical assertions (RLS, full BE flows) are *not* on the PR gate.

6. **Empty directories advertised as features.** `docs/adr/`, `scripts/codegen/`, `scripts/qa/`, `scripts/seed/`, `supabase/snippets/` are all empty but referenced in README. Either populate or remove.

7. **Wave-relative language survives in code and user docs.** `AuthContext.tsx`, `ProtectedRoute.tsx`, `tenant.ts`, and `docs/users/*` all carry "Wave N stops here / Wave N+1 will add" text that is now actively misleading. Periodic prose-purge needed.
